Claude Code LLM Self-Reflection Eval-Driven Auto-Versioning
Motivation

Skills decay.
They don't improve.

The current state

You write a SKILL.md. It runs. It makes a mistake. You manually edit it. Maybe. Next session you forget why you made that change. No history. No signal. No loop.

The missing piece

Every skill run contains signal — what worked, what didn't, what edge case appeared. That signal evaporates. Nothing captures it. Nothing learns from it.

The gap: LLM skills are static documents maintained by humans. There's no mechanism to accumulate knowledge from runs into the skill itself.
What This Is

A self-reflection
loop, built into every skill

A self-improving skill is a normal SKILL.md + scripts/ — wrapped with a feedback loop.


After each run, Claude checks its own output against the skill's evaluation criteria, identifies the most important gap, proposes a surgical fix to SKILL.md, and — if approved — updates itself, snapshots the old version, and logs what changed and why.


No manual scoring. No separate workflow step. No eval pack selection. The improvement is part of the skill run — fully automatic.
1
Skill runs — normal execution
2
Self-reflect — check output against eval criteria
3
Gap found? — propose one surgical edit
4
Approved? — snapshot → apply → log
5
No gap? — silent. No noise.

One edit at a time

Never a rewrite. One instruction clarified, one missing step added, one ambiguous constraint tightened. CHANGELOG entries stay meaningful — each traces to one real gap.

Silent when clean

A skill that ran well generates zero noise. The loop only surfaces when there's a real gap. Silence = signal of quality.

File System

Skill Directory Structure

~/.claude/skills/{skill-name}/
├── SKILL.md            core instructions, <300 lines
├── README.md           user-facing docs
│
├── eval/
│   ├── criteria.md     auto-populated on first run
│   └── examples.md     2-3 golden I/O pairs
│
├── feedback/
│   ├── improvements.md append-only improvement log
│   └── edge-cases.md   uncovered situations
│
├── versions/
│   ├── v1.md           full SKILL.md snapshot
│   ├── v2.md           snapshot before next edit
│   └── CHANGELOG.md    human-readable history
│
└── scripts/            execution scripts (unchanged)
!
versions/vN.md is written before the edit — never after. Rollback is always one cp command.
!
improvements.md is append-only — the skill's full learning history.
!
criteria.md is stable once set — improving the skill never moves the goalposts.
# rollback to any version
cp ~/.claude/skills/my-skill/versions/v2.md \
   ~/.claude/skills/my-skill/SKILL.md
Evaluation System

Eval Packs &
Auto-Selection

Criteria are built from two layers combined automatically on the first run against any new skill.


Universal Evals (always included)

Instruction-Following · Completeness · Accuracy · Hallucination · Safety/Risk · Constraint Adherence

Type-Specific Packs (11 packs, 141 evals)

Writing · Formatting/Conversion · Tool/API/Automation · Coding · Research · Multi-Turn · Agentic Workflow · Batch/Data · Self-Improvement · Security/Safety · Example/Retrieval

How auto-selection works

1
Read SKILL.md — description, output type, tools used
2
Match against type → pack table in eval-pack-selector.md
3
Copy pack + trim irrelevant evals → write to eval/criteria.md
4
Inform user: "Auto-selected Pack N: [type]"

One-time setup. No manual intervention. Manual override available if auto-selection is wrong (rare).
self-improving-instructions.md

The 7-Step Loop

1
Locate the skill
Find SKILL.md, criteria.md, edge-cases.md, versions/
2
Auto-select eval pack
If criteria.md has TODO → populate now, once only
3
Self-reflect
Pass / Partial / Miss against each criterion + edge-cases.md
4
Decide
No gap → silent. Gap → form one proposal with exact diff
5
Handle response
yes → apply · adjust → refine + apply · skip → log edge case
6
Apply the edit
Snapshot → edit → improvements.md → CHANGELOG.md → subagents
Step 7 — Confirm: "Updated to v(N+1). [One sentence on what changed.]" — nothing more.
Versioning

Full snapshots.
One-command rollback.

# improvements.md — machine-readable log
[2026-05-14] v1→v2 | WHAT: Added null-check step before
              batch API call | WHY: Completeness criterion

[2026-05-21] v2→v3 | WHAT: Tightened output format
              constraint to enforce JSON-only | WHY: user suggestion

[2026-05-28] v3→v4 | WHAT: Added edge case handling for
              empty sheet rows | WHY: edge-cases.md hit in live run

# CHANGELOG.md — human-readable
## v4 — 2026-05-28
- Added guard for empty rows in Google Sheets input
- Triggered by live run on 19-row dataset with 2 blanks
- Closes gap found in edge-cases.md from 2026-05-15

Why full snapshots, not diffs

The agent reads the full version history before proposing a new improvement. Full snapshots = trivial rollback + the agent can reason about the entire previous state, not reconstruct it from patches.

The agent reads history

Before proposing, it reads improvements.md to avoid re-proposing reverted changes and to surface recurring patterns across runs.

Quality Gates

Subagents run
after every edit

Subagent When invoked What it does
documenter After every accepted SKILL.md edit Verifies the edit is coherent — no contradictions introduced, no broken cross-references, no duplicate instructions
script-reviewer When scripts/ were modified during evolution Checks updated scripts for bugs, safety issues, and edge cases
qa Optional — when examples.md has executable inputs Runs golden examples as regression tests after each edit — confirms known-good cases still pass
Automatic. documenter and script-reviewer are invoked by self-improving-instructions directly — no manual calls needed. The skill protects its own coherence.
Onboarding

Wrapping any skill
in 30 seconds

One script creates the full directory structure, stubs every file, snapshots v1, and adds the self-improving footer to SKILL.md.

# Wrap an existing skill
bash import-skill.sh \
  ~/.claude/skills/my-skill/SKILL.md

# After first skill run
/self-improving-instructions

# Manual loop trigger (any time)
/self-improving-instructions my-skill

The only manual step after running the script: write 2–3 entries in eval/examples.md. Everything else is automatic.

When to create a skill at all

✓
Used 2+ times — not a one-off
✓
3+ steps or meaningful decisions involved
✓
Stable enough to document — workflow won't change completely next week

If simpler than 3 steps → add to CLAUDE.md or memory instead.

Design Decisions

What we deliberately
left out

Dropped Why
1–5 rating system Scores are vague — they don't tell the model what to fix. Self-reflection against specific criteria is actionable. Scores are not.
successes.md / failures.md Not useful without ratings. improvements.md and edge-cases.md carry the same signal more cleanly.
Manual eval pack selection Auto-selection on first run eliminates this entirely. One less thing to configure.
Simulation agent Powerful but complex. Build after the manual loop is battle-tested on several real skills. See Phase 2.
Install wizard / config file Overkill for personal tooling. The framework doc covers every decision.
Roadmap

Phase 2:
Simulation Agent

Not built yet. The architecture is defined — build this once the manual feedback loop has been validated on several real skills and the file structures are stable.


What it does

Generates synthetic test scenarios for any skill. Runs evals without needing a real user session. Auto-applies improvements when confidence is high enough.


Why this order matters

Manual loop first — validates the structure, builds intuition for what makes a good improvement proposal, surfaces edge cases in the loop itself. Then automate.

Current loop: Run → eval check → propose one edit → human approves → snapshot → apply → log
Phase 2 loop: Simulate → eval check → propose one edit → confidence threshold → auto-apply → log → notify human

The file structure is already Phase 2 ready. No changes needed to versions/, improvements.md, or CHANGELOG.md. The simulation agent writes to the same places.

The system

Automatic

Eval pack selected, criteria populated, subagents invoked — all without user setup. Run once, get a fully structured skill.

Surgical

Reversible

Full SKILL.md snapshots at every version. Rollback = one cp. Append-only logs mean history is never lost.
